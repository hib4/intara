## 📖 Overview

The **Intara Core API** is a production-grade, asynchronous RESTful backend powering the entire Intara SaaS platform. Built on **FastAPI** and **Python 3.11**, it exposes a secure, self-documenting API that drives two core product pillars:

1. **No-Code RAG Chatbot Engine** — Accepts uploaded PDF/TXT documents, chunks and vectorises their content using Alibaba Cloud DashScope embeddings (Qwen), persists the vectors in a **pgvector**-enabled PostgreSQL database, and serves contextually accurate responses via a full **Retrieval-Augmented Generation (RAG)** pipeline powered by **Qwen/QwQ** large language models.

2. **AI Financial Insight Assistant** — Processes uploaded CSV financial datasets and enables natural language querying, returning structured insights and summaries generated by the Qwen model.

All endpoints are protected with **JWT bearer token authentication** (PyJWT), schema-validated with **Pydantic v2**, and the database layer is fully managed with **SQLAlchemy 2.0 async** sessions and **Alembic** migrations.

## 🏗️ Architecture

The production deployment is hosted entirely on **Alibaba Cloud**, with a clean separation of concerns across three infrastructure tiers:

```
┌─────────────────────────────────────────────────────────┐
│                      Client (Browser)                   │
└─────────────────────┬───────────────────────────────────┘
                      │ HTTPS
┌─────────────────────▼───────────────────────────────────┐
│           Alibaba Cloud ECS (API Server)                │
│       FastAPI + Uvicorn  ·  Docker Container            │
│   • JWT Auth   • RAG Pipeline   • Document Processing   │
└──────────┬──────────────────────────────┬───────────────┘
           │ PostgreSQL (psycopg2)         │ HTTPS (DashScope SDK)
┌──────────▼──────────┐      ┌────────────▼────────────────┐
│  Alibaba ApsaraDB   │      │  Alibaba Cloud Model Studio  │
│  PostgreSQL + RDS   │      │  Qwen / QwQ LLM Inference    │
│  pgvector extension │      │  Text Embeddings (DashScope) │
└─────────────────────┘      └─────────────────────────────┘
```

| Tier             | Service                                  | Role                                             |
| ---------------- | ---------------------------------------- | ------------------------------------------------ |
| **API**          | Alibaba Cloud ECS                        | Runs the Dockerised FastAPI application          |
| **Database**     | Alibaba ApsaraDB for RDS (PostgreSQL 16) | Relational data + pgvector for similarity search |
| **AI Inference** | Alibaba Cloud Model Studio (DashScope)   | Qwen/QwQ LLM generation + text embeddings        |

| Feature                       | Description                                                                                                |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------- |
| 🔐 **JWT Authentication**     | Stateless, secure access/refresh token flow using PyJWT and bcrypt password hashing                        |
| 🤖 **RAG Pipeline**           | Full document ingestion → chunking (LangChain) → embedding → vector storage → retrieval → generation cycle |

| 🔄 **Alembic Migrations**     | Versioned, reproducible database schema management                                                         |
| 🐳 **Fully Dockerised**       | Single-command local environment with Docker Compose (API + pgvector DB)                                   |
| 📑 **Auto API Docs**          | Interactive Swagger UI and ReDoc generated automatically by FastAPI                                        |

## 🔑 Environment Variables

The API is configured via a `.env` file in the root of the `backend/` directory. Copy the example and populate the required values:

```bash
cp .env.example .env
```

| Variable                      | Required | Description                                                           | Example                                            |
| ----------------------------- | -------- | --------------------------------------------------------------------- | -------------------------------------------------- |
| `DATABASE_URL`                | ✅ Yes   | Full PostgreSQL connection string                                     | `postgresql://intara:intara_secret@db:5432/intara` |
| `SECRET_KEY`                  | ✅ Yes   | Long random string for JWT signing — **change in production**         | `openssl rand -hex 32`                             |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | No       | JWT token lifetime in minutes (default: `1440` = 24h)                 | `1440`                                             |
| `ALIBABA_API_KEY`             | ✅ Yes   | DashScope API key for Qwen LLM and embeddings                         | `sk-xxxxxxxxxxxxxxxxxxxx`                          |
| `GOOGLE_API_KEY`              | No       | Google Gemini API key (alternative embedding backend)                 | `AIzaSy...`                                        |
| `CHUNK_SIZE`                  | No       | Token chunk size for document splitting (default: `1000`)             | `1000`                                             |
| `CHUNK_OVERLAP`               | No       | Token overlap between chunks (default: `200`)                         | `200`                                              |
| `CORS_ORIGINS`                | No       | Comma-separated list of allowed frontend origins                      | `http://localhost:5173`                            |
| `PAYLABS_MERCHANT_ID`         | No       | PayLabs merchant identifier for payment processing                    | `MID-xxxxx`                                        |
| `PAYLABS_SECRET_KEY`          | No       | PayLabs server-side secret for webhook signature verification         | `sk_live_xxxx`                                     |
| `DEBUG`                       | No       | Enable FastAPI debug mode (default: `true` — **set `false` in prod**) | `false`                                            |

**`.env.example`**

```env
# ── Database ─────────────────────────────────────────────────
# For local Docker Compose, the host is 'db' (the service name)
# For production ApsaraDB, use the RDS connection string
DATABASE_URL=postgresql://intara:intara_secret@db:5432/intara

# ── Security ─────────────────────────────────────────────────
# Generate a strong secret: openssl rand -hex 32
SECRET_KEY=change-me-to-a-long-random-string-in-production
ACCESS_TOKEN_EXPIRE_MINUTES=1440

# ── Alibaba Cloud / DashScope (Qwen LLM + Embeddings) ────────
ALIBABA_API_KEY=

# ── Google AI (Optional alternative embedding backend) ───────
GOOGLE_API_KEY=

# ── RAG Chunking ─────────────────────────────────────────────
CHUNK_SIZE=1000
CHUNK_OVERLAP=200

# ── CORS ─────────────────────────────────────────────────────
# Comma-separated. In production, set to your production frontend URL
CORS_ORIGINS=http://localhost:5173

# ── Payments (PayLabs) ───────────────────────────────────────
PAYLABS_MERCHANT_ID=
PAYLABS_SECRET_KEY=

# ── Application ──────────────────────────────────────────────
DEBUG=true
```

## 🐳 Local Development (Dockerised)

The recommended and supported development workflow uses Docker Compose, which spins up both the FastAPI application and a `pgvector`-enabled PostgreSQL database.

### 1. Clone the Repository

```bash
git clone https://github.com/your-org/intara.git
cd intara/backend
```

### 2. Configure Environment

```bash
cp .env.example .env
# Populate ALIBABA_API_KEY and any other required keys
```

### 3. Build and Start All Services

```bash
docker compose up --build
```

> On the **first run**, Docker will build the `intara-api` image, pull the `pgvector/pgvector:pg16` image, and start both containers. The API container will wait for the database health check to pass before starting.

The API will be available at **[http://localhost:8000](http://localhost:8000)**.

### 4. Apply Database Migrations

In a **separate terminal**, run Alembic migrations to create all tables:

```bash
docker compose exec api alembic upgrade head
```

The application is now fully operational. 🎉

### 5. Stop All Services

```bash
docker compose down
```

To stop and **delete all data** (volumes):

```bash
docker compose down -v
```

## 🛠️ Makefile Shortcuts

A `Makefile` is provided for convenience. These targets wrap common Docker Compose and Alembic commands:

| Command                      | Description                                              |
| ---------------------------- | -------------------------------------------------------- |
| `make build`                 | Build Docker images                                      |
| `make up`                    | Start all services in detached mode                      |
| `make down`                  | Stop and remove containers and networks                  |
| `make restart`               | Full teardown → rebuild → start                          |
| `make logs`                  | Tail the API container logs                              |
| `make shell`                 | Open a bash shell inside the running API container       |
| `make db-shell`              | Open a `psql` session inside the database container      |
| `make clean`                 | Stop containers **and delete all volumes** (destructive) |
| `make migrate`               | Apply all pending Alembic migrations (`upgrade head`)    |
| `make migrate-gen msg="..."` | Auto-generate a new Alembic migration revision           |
| `make migrate-down`          | Roll back the last migration                             |
| `make migrate-current`       | Show the current Alembic revision                        |

## 🔄 Database Migrations (Alembic)

All schema changes are version-controlled via Alembic. Migration files live in `alembic/versions/`.

### Apply All Pending Migrations

```bash
# Via Make
make migrate

# Or directly
docker compose exec api alembic upgrade head
```

### Generate a New Migration (after changing `app/db/models.py`)

```bash
# Via Make
make migrate-gen msg="add subscription table"

# Or directly
docker compose exec api alembic revision --autogenerate -m "add subscription table"
```

### Roll Back the Last Migration

```bash
make migrate-down
# Or: docker compose exec api alembic downgrade -1
```

## 📑 API Documentation

FastAPI automatically generates interactive API documentation from the code and Pydantic schemas. No additional setup is required.

| Interface        | URL                                                                      | Description                                                        |
| ---------------- | ------------------------------------------------------------------------ | ------------------------------------------------------------------ |
| **Swagger UI**   | [http://localhost:8000/docs](http://localhost:8000/docs)                 | Interactive API explorer — try endpoints directly from the browser |
| **ReDoc**        | [http://localhost:8000/redoc](http://localhost:8000/redoc)               | Clean, readable documentation in the ReDoc format                  |
| **OpenAPI JSON** | [http://localhost:8000/openapi.json](http://localhost:8000/openapi.json) | Raw OpenAPI 3.x schema (useful for code generation)                |

### API Endpoint Groups

| Prefix           | Tag            | Description                                                 |
| ---------------- | -------------- | ----------------------------------------------------------- |
| `/api/auth`      | Authentication | Register, login, token management                           |
| `/api/chatbots`  | Chatbots       | CRUD operations for chatbot configurations                  |
| `/api/documents` | Documents      | Upload, list, and delete RAG knowledge-base documents       |
| `/api/chat`      | Public Chat    | Public-facing chat endpoint consumed by the embedded widget |

> **Note:** All endpoints except `/api/auth/register` and `/api/auth/login` require a valid `Authorization: Bearer <token>` header.

## 🔒 Security Considerations

- **Change `SECRET_KEY`** before any deployment. Generate a cryptographically strong value: `openssl rand -hex 32`
- **Set `DEBUG=false`** in production to disable stack traces in error responses
- **Restrict `CORS_ORIGINS`** to your exact production frontend domain
- **Never commit `.env`** to version control — it is listed in `.gitignore`
- All passwords are hashed using **bcrypt** via `passlib` before storage
